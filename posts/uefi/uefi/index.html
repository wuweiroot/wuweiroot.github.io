    X86 BIOS启动模式#

    • 先了解 X86 架构的工作模式， X86 有很多种工作模式： 但是对BIOS 来说我们只关注实模式、 保护模式、 SMM 模式。模式切换大概就是从实模式 –>保护模式（ 32bit） –>保护模式（ 32bit flat 模式） –>保护模式(X64 flat);
    • 在 POST 的过程中，在不同的阶段 cpu 会工作在不同的模式，一般来说在 SEC 阶段是实模式，在 PEI阶段是 32bit flat 模式，而在 DXE 及后续阶段是保护模式的 X64 flat 模式。

    UEFI启动的七个阶段#

    SEC PEI DXE BDS TSL RT AL

    SEC阶段#

    • 接收和处理系统上电或者重启信号
    • RestVector 的初始化，以及异常向量的初始化
    • SecCoreStartupWithStack 初始化 IDT
    • 初始化临时存储区域 cache作为内存
    • 传递系统参数给下一个阶段
    • 从实模式到保护模式，从32位模式到64位模式

    PEI阶段#

    • PEI内核调度PEIM
    • CPU初始化 内存初始化 芯片初始化
    • CPU 工作模式的切换
    • 初始化内存

    DXE阶段#

    • 初始化控制台设备（）
    • 加载必要的设备驱动
    • 初始化 CPU 的 GDT。
    • 初始化 CPU 的 IDT.
    • 根据系统设置加载和执行启动项

    BDS阶段#

    • 初始化控制台设备（输出输出设备）
    • 加载必要的设备驱动
    • 根据系统设置加载和执行启动项

    TSL阶段#

    • 是OS loder加载的第一阶段
    • 启动服务进入ExitBootServices服务调用后进入 Run time阶段

    RT阶段#

    • 系统控制器转移给OS Loder

    UEFI 内核数据结构

    IHANDLE#

    协议句柄列表

    • EFI_HANDLE是指向某种对象的指针，UEFI用它来表示某个对象。
    • UEFI扫描总线后，会为每个设备建立一个Controller对象，用于控制设备，所有该设备的驱动以Protocol的形式安装到这个Controller中，这个Controller对象就是一个EFI_HANDLE对象。
    • 当我们将一个.efi文件加载到内存中时，UEFI也会为改文件建立一个Image对象，这个Image对象也是一个EFI_HANDLE对象。在UEFI内部，EFI_HANDLE被理解为IHANDLE.

    PROTOCOL_ENTRY#

    每个不同的协议在协议数据库中都有1个条目。列出了支持此协议的每个处理程序以及已注册通知的列表。

    PROTOCOL_INTERFACE#

    安装在句柄上的每个协议都使用协议接口结构进行跟踪

    联系#

    • 一个ihandle可以挂多个 PROTOCOL_INTERFACE

      一个设备有多个协议, 如同一个PCIE设备可以支持DevicePath协议和PciIo协议

    • 一个 PROTOCOL_ENTRY 下可以挂多个 PROTOCOL_INTERFACE

      一个协议可以有多个设备， 如同一个Pcio协议可以安装到不同的PCIE设备身上

    • PROTOCOL_INTERFACE 是不会重复的。

      PROTOCOL_INTERFACE数量 等于 设备*协议,一个设备一个协议唯一指定一个PROTOCOL_INTERFACE

    UEFI代码架构#

    Package#

    • 以XXX.Pkg命名的文件夹
    • 一组模块和平台描述文件(dsc文件) 和包声明文件(dec文件)的集合
    • dsc文件用来指导各个模块的执行顺序。

    Module#

    • EDK2 每个工程模块由.inf文件和源文件组成
    • inf文件类似Linux的Makefile文件，用于指导源文件编译。

    UEFI 模块类型#

    • UEFI 驱动，符合UEFI驱动模型的驱动。

    • DXE驱动，DXE环境下运行的驱动。不遵循UEFI驱动模型

    • DXE SMM驱动，系统管理模式的驱动， 模块被加载到系统管理内存区，系统进入运行期改驱动依然有效。
