    列表#

    在RTOS中，列表通常用于管理任务、消息或事件的集合。可以是就绪任务列表、延时任务列表等

    列表项#

    列表中的每一个任务、消息或事件都是一个列表项，包含相关的信息，如任务状态、优先级等。

    就绪任务列表#

    存储所有已准备好运行的任务。RTOS调度器会从此列表中选择任务进行执行。

    延时任务列表#

    存储那些在未来某个时间点需要执行的任务。这些任务在达到预定时间时会被移动到就绪任务列表中。

    挂起任务列表#

    任务体都不能是无限循环的，必须调用能将自己挂起的函数 ，比如 vTaskDelay()。 调用能将任务挂起的函数中，都会先将任务从就绪列表删除，

    删除任务列表#

    任务状态#

    FreeRTOS 系统中的每一任务都有多种运行状态。系统初始化完成后，创建的任务就可 以在系统中竞争一定的资源，由内核进行调度

    就绪（ Ready）#

    任务在就绪列表中， 就绪的任务已经具备执行的能力，只等待调度器进行调度，新创建的任务会初始化为就绪态

    给当前任务TCB设置延时时间，切换任务

    • 时钟中断中做处理

    遍历就绪任务中的延时时间，非零则减少1

    • 任务转换函数中处理

    判断就绪列表中任务延时时间变量是否为0，不为0不调度

    运行（Running）#

    • 该状态表明任务正在执行， 此时它占用处理器， FreeRTOS 调度器选择运行的永远是处于最高优先级的就绪态任务，当任务被运行的一刻，它的任务状态就变成了运行态
    • 就绪列表中优先级最高的那个

    阻塞（Blocked）#

    • 如果任务当前正在等待某个时序或外部中断，我们就说这个任务处于阻塞状态，该任务不在就绪列表中。
    • 包含任务被挂起、 任务被延时、任务正在等待信号量、读写队列或者等待读写事件等

    通过阻塞列表实现阻塞延时#

    • 实现任务延时函数

      • 将当前任务插入延时列表（prvAddCurrentTaskToDelayedList）
        • prvAddCurrentTaskToDelayedList
          • 将任务从就绪列表中移除
          • 计算延时到期时，系统时基计数器xTickCount的值是多少 xTimeToWake = xConstTickCount + xTicksToWait;
          • 将延时到期的值设置为节点的排序值(xStateListItem)
          • 判断xTimeToWake是否溢出
            1. 溢出：
            • 任务按照升序加入溢出延时列表
            1. 不溢出：
            • 任务加入升序延时列表
            • 如果xTimeToWake小于xNextTaskUnblockTime ，更新下一个任务解锁时刻变量 xNextTaskUnblockTime的值

    • 时钟中断中处理

      • 判断当前时基计数器xTickCount是否溢出
        1. 溢出：
        • 交换 pxDelayedTaskList ，pxOverflowDelayedTaskList指针指向
        • 获取下一任务解锁时间 xNextTaskUnblockTime
          • 延时列表第一个成员得到他的排序值，即延时到期值，即下一个任务的解锁时间
        1. 不溢出：
        • 发现有下一延时任务到时 xConstTickCount >= xNextTaskUnblockTime
          • 出栈所有到时任务
          • 移除延时列表，根据优先级加入，加入同一优先级就绪列表尾部
          • 剩下第一个阻塞列表中的任务为下一个到期任务，更新下一个任务解锁时刻变量 xNextTaskUnblockTime
      • 如果新加入的任务优先级高于当前任务则任务切换

    挂起态(Suspended)：#

    处于挂起态的任务对调度器而言是不可见的， 让一个任务进入挂起状态的唯一办法就是调用 vTaskSuspend()函数；

    我们可以这么理解挂起态与阻塞态的区别，当任务有较长的时间不允许运行 的时候，我们可以挂起任务，这样子调度器就不会管这个任务的任何信息，直到 我们调用恢复任务的 API 函数；而任务处于阻塞态的时候，系统还需要判断阻塞 态的任务是否超时，是否可以解除阻塞。

    任务切换#

    在RTOS中，任务切换是指从一个任务切换到另一个任务的过程。RTOS通常会使用时间片轮转或优先级调度来实现任务切换。

    中断函数#

    • PendSV PendSV是可以挂起的异常，它和SVC协同使用，顾名思义，PendSV和SVC不同，PendSV可以像普通中断一样被挂起而延迟执行，OS可以利用该特点，等待其他重要任务处理完成后才执行该异常。挂起PendSV的方法是：往NVIC的PendSV挂起寄存器写1。
    • SVC SVC用于产生调用系统函数的请求，例如操作系统不允许用户应用程序直接访问操作硬件设备，而是通过一些系统服务函数操作硬件设备，当用户程序想要访问某个硬件时，首先触发一个SVC异常，发出对系统服务函数的调用请求，然后操作系统提供的SVC异常服务函数得到执行，该函数再调用操作硬件的系统函数，最终间接访问硬件设备

    切换过程 ->任务A->任务B->任务A#

    首次执行任务 >任务A#

    只切换栈不保存上下文

    • 通过触发 SVC_Handler 进入内核中断。
    • 通过当前pcb指针找到当前的pcd中任务栈栈顶指针，栈顶指针值存入R0寄存器
    • 通过R0寄存器将任务栈中的R4-R11依次出栈填充CPU寄存器，同时R0寄存器随出栈过程递增
    • 将R0寄存器当前指向任务栈的位置赋值给psp寄存器。从而改变CPU执行方向
    • 通过bx r14指令返回，psp指向的栈中内容依次弹出，其中的R15(PC)决定CPU执行方向。此时的R15来自任务创建时填充的线程函数地址。

    任务A->任务B#

    保存任务A的上下文环境#
    • 通过触发 PendSV_Handler 进入内核中断
    • 此时xPSR，PC（R15），R14，R12，R3，R2，R1，R0（任务的形参）这些CPU寄存器的值会自动保存到任务的栈中
    • 此时栈中PC指针指向进入中断前函数执行位置。
    • 通过当前pcb指针找到当前的pcd中任务栈栈顶指针，栈顶指针值存入R2寄存器
    • 当前psp指向的栈顶位置赋值给R0寄存器
    • 将CPU r4-r11的寄存器依次复制给R0指向的栈中，R0同时递增。最终指向存放R4的栈位置
    • 将此时R0的值赋值到TCB中的栈顶指针成员，完成任务A栈顶指针更新，完成任务A上下文环境保存。
    切换到任务B的上下文环境#
    • 调用vTaskSwitchContext将变量pxCurrentTCB指向新的任务
    • 通过当前pcb指针找到当前的pcd中任务栈栈顶指针，栈顶指针值存入R0寄存器
    • 通过R0寄存器将任务栈中的R4-R11依次出栈填充CPU寄存器，同时R0寄存器随出栈过程递增
    • 将R0寄存器当前指向任务栈的位置赋值给psp寄存器。从而改变CPU执行方向
    • 通过bx r14指令返回，psp指向的栈中内容依次弹出，其中的R15(PC)决定CPU执行方向。此时的R15来自任务创建时填充的线程函数地址。

    保存任务B的上下文环境#
    • 通过触发 PendSV_Handler 进入内核中断
    • 此时xPSR，PC（R15），R14，R12，R3，R2，R1，R0（任务的形参）这些CPU寄存器的值会自动保存到任务的栈中
    • 此时栈中PC指针指向进入中断前函数执行位置。
    • 通过当前pcb指针找到当前的pcd中任务栈栈顶指针，栈顶指针值存入R2寄存器
    • 当前psp指向的栈顶位置赋值给R0寄存器
    • 将CPU r4-r11的寄存器依次复制给R0指向的栈中，R0同时递增。最终指向存放R4的栈位置
    • 将此时R0的值赋值到TCB中的栈顶指针成员，完成任务B栈顶指针更新，完成任务B上下文环境保存。
    切换到任务A的上下文环境#
    • 调用vTaskSwitchContext将变量pxCurrentTCB指向新的任务
    • 通过当前pcb指针找到当前的pcd中任务栈栈顶指针，栈顶指针值存入R0寄存器
    • 通过R0寄存器将任务栈中的R4-R11依次出栈填充CPU寄存器，同时R0寄存器随出栈过程递增
    • 将R0寄存器当前指向任务栈的位置赋值给psp寄存器。从而改变CPU执行方向

    TCB#

    TCB（任务控制块）是实时操作系统（RTOS）中用于管理任务的数据结构。每个任务在RTOS中都有一个对应的TCB，用于存储任务的状态、优先级、堆栈指针等信息。以下是TCB的一些重要信息：

    优先级#

    RTOS中的任务通常具有优先级，优先级高的任务会先于优先级低的任务执行。调度算法会根据优先级来选择执行的任务

    时间片#

    在RTOS中，时间片是分配给每个任务的执行时间段。时间片结束后，RTOS会进行任务切换，允许其他任务获得CPU资源。

    队列#

    队列又称消息队列，是一种常用于任务间通信的数据结构

    • 消息队列是一种异步的通信方式

    创建队列#

    创建消息队列时 FreeRTOS 会先给消息队列分配一块内存空间

    • 这块内存的大小等于消息队列控制块大小加上（单个消息空间大小与消息队列长度的乘积）

    • FreeRTOS 的消息队列控制块由多个元素组成，当消息队列被创建时，系统会为控制块分配对应的内存空间，用于保存消息队列的一些信息如消息的存储位置，头指针 pcHead、尾指针 pcTail、消息大小 uxItemSize 以及队列长度 uxLength 等。同时每个消息队列都与消息空间在同一段连续的内存空间中，

    • 创建成功的时候就已经分配好每个消息空间与消息队列的容量，无法更改，每个消息空间可以存放不大于消息大小 uxItemSize 的任意类型的数据，

    API#

    /* 创建队列: 长度为5，数据大小为4字节(存放一个整数) */
xQueue = xQueueCreate( 5, sizeof( int32_t ) );

    写队列#

    API#

    /* 写队列
* xQueue: 写哪个队列
* &lValueToSend: 写什么数据? 传入数据的地址, 会从这个地址把数据复制进队列
* 0: 不阻塞, 如果队列满的话, 写入失败, 立刻返回
*/
xStatus = xQueueSendToBack( xQueue, &lValueToSend, 0 );

    队列集#

    队列集的本质也是队列，存放的内容是队列的句柄
